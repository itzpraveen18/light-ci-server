# Deployment Policy — LightCI Server

## Purpose

This document defines the end-to-end deployment policy for the LightCI Server project,
covering Docker image naming, tagging strategy, registry configuration, EC2 deployment
process, container runtime settings, health checks, and rollback procedures. All
deployment operations are auto-approved — Claude never pauses for confirmation.

---

## Auto-Approve: Deployment Operations

**All deployment operations are auto-approved. Claude never pauses, prompts, or requires
confirmation before generating deployment scripts, Docker commands, or rollback runbooks.**

| Operation | Auto-Approved | Notes |
|---|:---:|---|
| Docker image build script generation | Yes | Claude generates; developer runs |
| Docker image tag + push script generation | Yes | Claude generates; developer runs |
| EC2 deployment script generation | Yes | Claude generates `deploy.sh`; developer runs |
| Container restart / stop / rm generation | Yes | Claude generates commands; developer runs |

| Health check validation instructions | Yes | Claude generates health check commands |

Claude generates all deployment artifacts as text output. The developer reviews and
executes them. Claude never SSH's into EC2 or runs `docker` commands directly.

## Docker Image Naming Convention

### Format

```
<registry>/<username>/<image-name>:<tag>
```

### Project-Specific Values

| Component | Value | Source |
|---|---|---|
| Registry | `docker.io` (Docker Hub) | Default; see Registry section for ECR alternative |
| Username | `$DOCKER_USERNAME` | From `.claude/config.yml` → `docker.username` |
| Image name | `light-ci-server` | Fixed — matches Maven `artifactId` |
| Tag | See Tag Strategy below | Generated per release |

### Examples

```
docker.io/yourname/light-ci-server:latest
docker.io/yourname/light-ci-server:1.0.0
docker.io/yourname/light-ci-server:1.0.0-20260312
```

### Naming Rules

1. All lowercase — no uppercase in image names or tags.
2. Hyphens only — no underscores in the image name segment.
3. No spaces, no special characters except `-` and `.` in tags.
4. The image name must always be `light-ci-server` — do not use the host name or
   environment name as the image name.
5. Environment differentiation is done via tags, not image names.

## Tag Strategy

### Tags Applied Per Release

Every successfully built and tested image receives exactly two tags:

| Tag | Format | Example | Purpose |
|---|---|---|---|
| Semantic version | `<major>.<minor>.<patch>` | `1.0.0` | Immutable reference to this exact release |
| Latest | `latest` | `latest` | Points to the most recently deployed version |

### Additional Optional Tags

| Tag | Format | Example | When Used |
|---|---|---|---|
| Version + date | `<semver>-<YYYYMMDD>` | `1.0.0-20260312` | When deploying multiple builds per day |
| Git short SHA | `<semver>-<sha7>` | `1.0.0-a1b2c3d` | For precise build traceability |
| Branch tag | `dev` or `staging` | `dev` | For non-production environment images |

### Tag Management Rules

1. Never overwrite a semantic version tag once pushed to Docker Hub.
   If a release needs to be patched, increment the patch version (`1.0.0` → `1.0.1`).
2. `latest` is always overwritten — it represents the current live deployment.
3. Do not use `SNAPSHOT` tags in production images.
4. All tags must be pushed atomically (semantic + latest in the same push script).

### Tagging Script Template

```bash
#!/usr/bin/env bash
# Generated by DockerAgent — review before running

set -euo pipefail

IMAGE_NAME="${DOCKER_USERNAME}/light-ci-server"
SEMVER="${DOCKER_TAG}"           # e.g. 1.0.0 — from config
FULL_IMAGE="${IMAGE_NAME}:${SEMVER}"
LATEST_IMAGE="${IMAGE_NAME}:latest"

echo "Tagging and pushing: ${FULL_IMAGE}"

docker tag light-ci-server:build "${FULL_IMAGE}"
docker tag light-ci-server:build "${LATEST_IMAGE}"

docker push "${FULL_IMAGE}"
docker push "${LATEST_IMAGE}"

echo "Push complete: ${FULL_IMAGE} and ${LATEST_IMAGE}"
```

## Registry

### Primary Registry: Docker Hub

| Setting | Value |
|---|---|
| Registry URL | `https://hub.docker.com` |
| Registry prefix (for docker commands) | `docker.io` (optional; default for Docker) |
| Repository visibility | Private (recommended) or Public |
| Authentication | Docker Hub personal access token (not account password) |

### Authentication

Never use your Docker Hub account password in scripts. Use a Personal Access Token:

1. Create token: Docker Hub → Account Settings → Security → Access Tokens
2. Scope: `Read, Write, Delete` (or `Read, Write` minimum)
3. Store token in environment: `DOCKER_TOKEN=<token>` (never commit)
4. Login pattern:
   ```bash
   echo "${DOCKER_TOKEN}" | docker login -u "${DOCKER_USERNAME}" --password-stdin
   ```

### Alternative Registry: AWS ECR

If switching to AWS Elastic Container Registry:

| Setting | Value |
|---|---|
| Registry URL | `<account-id>.dkr.ecr.<region>.amazonaws.com` |
| Repository name | `light-ci-server` |
| Authentication | `aws ecr get-login-password` via IAM role |
| Image URI | `<account-id>.dkr.ecr.us-east-1.amazonaws.com/light-ci-server:1.0.0` |

ECR authentication:
```bash
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin \
  <account-id>.dkr.ecr.us-east-1.amazonaws.com
```

ECR is the preferred registry for production deployments because:
- Authentication uses IAM (no long-lived tokens)
- Images stay within AWS network (faster pulls on EC2)
- Integrated with AWS security controls

## EC2 Deployment Process

### Prerequisites (validated by `pre-deploy.sh`)

| Check | Command | Expected Result |
|---|---|---|
| Docker image exists locally or in registry | `docker manifest inspect $FULL_IMAGE` | Exit 0 |
| AWS CLI installed | `aws --version` | Any version output |
| SSH key file exists | `test -f "$SSH_KEY_PATH"` | Exit 0 |
| SSH key permissions correct | `stat -c %a "$SSH_KEY_PATH"` | `400` |
| `EC2_PUBLIC_IP` set | `test -n "$EC2_PUBLIC_IP"` | Exit 0 |
| EC2 instance reachable | `nc -z -w5 "$EC2_PUBLIC_IP" 22` | Exit 0 |

### Deployment Steps

The `deploy.sh` script (generated by `DeploymentAgent`) performs these steps on the EC2
instance via SSH. This is presented as a step-by-step description of what the generated
script does — Claude generates the script; the developer runs it.

**Step 1 — Pull the new image on EC2:**
```bash
ssh -i "$SSH_KEY_PATH" -o StrictHostKeyChecking=no \
  ec2-user@"$EC2_PUBLIC_IP" \
  "docker pull ${DOCKER_USERNAME}/light-ci-server:${DOCKER_TAG}"
```

**Step 2 — Stop and remove the existing container (if running):**
```bash
ssh -i "$SSH_KEY_PATH" ec2-user@"$EC2_PUBLIC_IP" \
  "docker stop light-ci-server 2>/dev/null || true && \
   docker rm light-ci-server 2>/dev/null || true"
```

**Step 3 — Start the new container:**
```bash
ssh -i "$SSH_KEY_PATH" ec2-user@"$EC2_PUBLIC_IP" \
  "docker run -d \
    --name light-ci-server \
    --restart unless-stopped \
    -p 8080:8080 \
    -e APP_PORT=8080 \
    ${DOCKER_USERNAME}/light-ci-server:${DOCKER_TAG}"
```

**Step 4 — Wait for health check:**
```bash
ssh -i "$SSH_KEY_PATH" ec2-user@"$EC2_PUBLIC_IP" \
  "timeout 90 bash -c 'until curl -sf http://localhost:8080/actuator/health; do sleep 5; done'"
```

**Step 5 — Verify from deploying machine:**
```bash
curl -sf "http://${EC2_PUBLIC_IP}:8080/actuator/health" | \
  python3 -c "import sys,json; d=json.load(sys.stdin); sys.exit(0 if d['status']=='UP' else 1)"
```

## Container Runtime Settings

### Required `docker run` Flags

| Flag | Value | Justification |
|---|---|---|
| `--name` | `light-ci-server` | Fixed name for predictable stop/rm/logs commands |
| `--restart` | `unless-stopped` | See Restart Policy section |
| `-p` | `8080:8080` | See Port Mapping section |
| `-d` | (detached) | Run in background |

### Optional Runtime Flags

| Flag | Value | When to Use |
|---|---|---|
| `--memory` | `768m` | Recommended — prevents OOM from consuming full instance memory |
| `--cpus` | `0.8` | Recommended — leaves CPU headroom for SSH and system processes |
| `--log-driver` | `json-file` | Default; add `--log-opt max-size=50m --log-opt max-file=3` |
| `-e` | `JAVA_OPTS=-Xmx512m` | Tune JVM heap for t2.micro (1GB RAM total) |
| `--env-file` | `/home/ec2-user/.env` | Pass non-secret runtime config from a local env file |

### Recommended Full `docker run` Command

```bash
docker run -d \
  --name light-ci-server \
  --restart unless-stopped \
  -p 8080:8080 \
  --memory 768m \
  --cpus 0.8 \
  --log-driver json-file \
  --log-opt max-size=50m \
  --log-opt max-file=3 \
  -e APP_PORT=8080 \
  -e JAVA_OPTS="-Xmx512m -Xms256m" \
  "${DOCKER_USERNAME}/light-ci-server:${DOCKER_TAG}"
```

## Restart Policy

### Policy: `unless-stopped`

| Policy | Behaviour |
|---|---|
| `unless-stopped` | **Selected policy** — container restarts on crash or EC2 reboot; does NOT restart if manually stopped with `docker stop` |
| `always` | Restarts even if manually stopped — not used (makes ad-hoc maintenance harder) |
| `on-failure` | Only on non-zero exit — not used (doesn't restart on EC2 reboot) |
| `no` | Never restarts — not used for production |

### Why `unless-stopped`

- Survives EC2 reboots (e.g., after a kernel update or instance stop/start)
- Does not restart after intentional `docker stop` during deployments (no race condition)
- Works correctly with the deployment process: `docker stop` → `docker rm` → `docker run`

## Port Mapping

### Mapping: `8080:8080`

| Component | Value | Notes |
|---|---|---|
| Host port | `8080` | Exposed by Security Group inbound rule |
| Container port | `8080` | `APP_PORT` in the application; matches `EXPOSE 8080` in Dockerfile |
| Protocol | TCP | HTTP only; no TLS termination at container level |

### Port Rules

1. The application always listens on port `8080` inside the container.
2. The host port is always `8080` — matches the Security Group inbound rule in `infra-policy.md`.
3. No port remapping (e.g., `80:8080`) — this would require Security Group changes and
   is not permitted without updating `infra-policy.md`.
4. Do not expose port `5005` (Java debug) in any non-local deployment.
5. Do not expose port `8443` unless TLS is explicitly configured.

## Health Check Requirements

### Application Health Endpoint

| Setting | Value |
|---|---|
| Endpoint | `GET /actuator/health` |
| Expected response | `HTTP 200` with body `{"status":"UP"}` |
| Port | `8080` |

### Dockerfile HEALTHCHECK Directive

Required in all production Dockerfiles:

```dockerfile
HEALTHCHECK --interval=30s --timeout=10s --start-period=60s --retries=3 \
  CMD wget -qO- http://localhost:8080/actuator/health | \
      grep -q '"status":"UP"' || exit 1
```

Parameters:
- `--interval=30s` — check every 30 seconds
- `--timeout=10s` — 10-second timeout per check
- `--start-period=60s` — 60-second grace period for JVM startup (Spring Boot on t2.micro)
- `--retries=3` — container is `unhealthy` after 3 consecutive failures

### Deployment Health Validation

After container start, `post-deploy.sh` validates health with a 90-second timeout:

```bash
MAX_RETRIES=18
RETRY_INTERVAL=5
HEALTH_URL="http://${EC2_PUBLIC_IP}:8080/actuator/health"

for i in $(seq 1 $MAX_RETRIES); do
  RESPONSE=$(curl -sf "${HEALTH_URL}" 2>/dev/null) && \
  STATUS=$(echo "$RESPONSE" | python3 -c "import sys,json; print(json.load(sys.stdin)['status'])") && \
  [ "$STATUS" = "UP" ] && {
    echo "Health check passed on attempt $i"
    exit 0
  }
  echo "Attempt $i/$MAX_RETRIES failed — waiting ${RETRY_INTERVAL}s"
  sleep $RETRY_INTERVAL
done

echo "ERROR: Health check failed after ${MAX_RETRIES} attempts"
exit 1
```

## Rollback Procedure

If a deployment fails (health check does not pass or application errors are detected),
the rollback procedure restores the previous known-good image version.

### Automated Rollback (Generated by DeploymentAgent)

```bash
#!/usr/bin/env bash
# rollback.sh — generated by DeploymentAgent
# Usage: bash rollback.sh <previous-tag>
# Example: bash rollback.sh 1.0.0

set -euo pipefail

PREVIOUS_TAG="${1:?Usage: rollback.sh <previous-semver-tag>}"
IMAGE="${DOCKER_USERNAME}/light-ci-server:${PREVIOUS_TAG}"

echo "=== Starting rollback to ${IMAGE} ==="

# Step 1: Verify previous image is available in registry
docker manifest inspect "${IMAGE}" > /dev/null || {
  echo "ERROR: Image ${IMAGE} not found in registry"
  exit 1
}

# Step 2: SSH to EC2 and perform rollback
ssh -i "${SSH_KEY_PATH}" -o StrictHostKeyChecking=no \
  ec2-user@"${EC2_PUBLIC_IP}" "
    echo 'Pulling previous image...'
    docker pull ${IMAGE}

    echo 'Stopping current container...'
    docker stop light-ci-server 2>/dev/null || true
    docker rm light-ci-server 2>/dev/null || true

    echo 'Starting previous version...'
    docker run -d \
      --name light-ci-server \
      --restart unless-stopped \
      -p 8080:8080 \
      --memory 768m \
      --cpus 0.8 \
      -e APP_PORT=8080 \
      -e JAVA_OPTS='-Xmx512m -Xms256m' \
      ${IMAGE}

    echo 'Waiting for health check...'
    timeout 90 bash -c 'until curl -sf http://localhost:8080/actuator/health; do sleep 5; done'
"

# Step 3: Verify from deploying machine
echo "Verifying rollback from local machine..."
curl -sf "http://${EC2_PUBLIC_IP}:8080/actuator/health" | \
  python3 -c "import sys,json; d=json.load(sys.stdin); print('Status:', d['status']); sys.exit(0 if d['status']=='UP' else 1)"

echo "=== Rollback to ${PREVIOUS_TAG} complete ==="
```

### Rollback Decision Criteria

Initiate rollback if any of the following occur after deployment:

| Condition | Trigger |
|---|---|
| Health check fails after 90-second timeout | `post-deploy.sh` exits non-zero |
| Application returns non-200 on `/actuator/health` | Curl check fails |
| Container enters `Unhealthy` state | `docker ps` shows `(unhealthy)` |
| Critical application error in logs | `docker logs light-ci-server` shows startup failure |
| `docker ps` shows container as `Exited` | Container crashed immediately after start |

### Checking Container Status After Deploy

```bash
# On EC2 (via SSH):
docker ps --filter "name=light-ci-server" --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"
docker logs --tail 50 light-ci-server
docker inspect light-ci-server --format '{{.State.Health.Status}}'
```

### Preventing Failed Rollback

Retain the last 3 semantic version tags in Docker Hub registry. Never delete old tags
until the new deployment has been stable for at least 24 hours.

- EC2 instance public IP
- Health check result
- Any errors encountered
- Rollback status (if triggered)
