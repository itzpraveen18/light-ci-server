# AWSInfraAgent

## Purpose

AWSInfraAgent generates complete, production-ready Terraform HCL files to provision
AWS EC2 infrastructure for the LightCI Server deployment. It creates a security group
with the correct inbound/outbound rules, provisions an EC2 instance with a Docker
bootstrap `user_data` script, and outputs the public IP and application URL.

AWSInfraAgent NEVER runs `terraform apply` or any AWS CLI command that mutates
infrastructure. It generates files only. The developer reviews and applies them.

## Security Group Rules Generated

### Inbound Rules

| Rule | Protocol | Port | Source | Reason |
|---|---|---|---|---|
| SSH access | TCP | 22 | `0.0.0.0/0` | Developer SSH access (restrict to your IP in production) |
| Application | TCP | 8080 | `0.0.0.0/0` | LightCI Server web UI |
| HTTPS (optional) | TCP | 443 | `0.0.0.0/0` | If TLS termination is added |

### Outbound Rules

| Rule | Protocol | Port | Destination | Reason |
|---|---|---|---|---|
| All outbound | All | All | `0.0.0.0/0` | Docker pull, package manager, AWS APIs |

## Step-by-Step Plan

### Step 1 — Read existing Terraform files
AWSInfraAgent reads `terraform/main.tf`, `terraform/variables.tf`, and
`terraform/outputs.tf` to understand current state before generating updates.

### Step 2 — Determine AMI
AWSInfraAgent uses a Terraform `data` source to always select the latest
Amazon Linux 2023 AMI automatically — no hardcoded AMI IDs.

```hcl
data "aws_ami" "amazon_linux_2023" {
  most_recent = true
  owners      = ["amazon"]
  filter {
    name   = "name"
    values = ["al2023-ami-2023.*-x86_64"]
  }
}
```

### Step 3 — Generate main.tf
Includes: provider, data sources (VPC, subnet, AMI), security group, EC2 instance
with `user_data` that installs Docker on first boot.

### Step 4 — Generate variables.tf
All inputs declared as variables with types, descriptions, and sensible defaults.

### Step 5 — Generate outputs.tf
Outputs: `instance_public_ip`, `app_url`, `ssh_command` for easy access after apply.

### Step 6 — Generate plan summary
AWSInfraAgent produces a text summary of what `terraform apply` would create:
- 1 x Security Group with 2 inbound rules
- 1 x EC2 instance (type, AMI, region)
- Estimated AWS cost at free-tier vs on-demand pricing

## Generated Files

### terraform/main.tf

```hcl
# Generated by AWSInfraAgent — LightCI Server Infrastructure
# Review carefully before running: terraform init && terraform plan

terraform {
  required_version = ">= 1.5.0"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region  = var.region
  profile = var.aws_profile
}

# --- Data Sources ---

data "aws_vpc" "default" {
  default = true
}

data "aws_subnet" "default" {
  vpc_id            = data.aws_vpc.default.id
  default_for_az    = true
  availability_zone = "${var.region}a"
}

data "aws_ami" "amazon_linux_2023" {
  most_recent = true
  owners      = ["amazon"]
  filter {
    name   = "name"
    values = ["al2023-ami-2023.*-x86_64"]
  }
  filter {
    name   = "virtualization-type"
    values = ["hvm"]
  }
}

# --- Security Group ---

resource "aws_security_group" "lightci_sg" {
  name        = "lightci-server-sg"
  description = "Security group for LightCI Server EC2 instance"
  vpc_id      = data.aws_vpc.default.id

  ingress {
    description = "SSH access"
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    description = "LightCI application port"
    from_port   = var.app_port
    to_port     = var.app_port
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  egress {
    description = "All outbound traffic"
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name        = "lightci-server-sg"
    Project     = "LightCI"
    Environment = var.environment
    ManagedBy   = "Terraform"
  }
}

# --- EC2 Instance ---

resource "aws_instance" "lightci_server" {
  ami                    = data.aws_ami.amazon_linux_2023.id
  instance_type          = var.instance_type
  key_name               = var.key_pair_name
  subnet_id              = data.aws_subnet.default.id
  vpc_security_group_ids = [aws_security_group.lightci_sg.id]

  root_block_device {
    volume_size           = 20
    volume_type           = "gp3"
    delete_on_termination = true
    encrypted             = true
  }

  user_data = <<-EOF
    #!/bin/bash
    set -e
    dnf update -y
    dnf install -y docker
    systemctl enable docker
    systemctl start docker
    usermod -aG docker ec2-user
    echo "Docker installed and running" > /tmp/docker-ready.txt
  EOF

  tags = {
    Name        = "lightci-server"
    Project     = "LightCI"
    Environment = var.environment
    ManagedBy   = "Terraform"
  }
}
```

```hcl

  description = "Public IP address of the LightCI Server EC2 instance"
  value       = aws_instance.lightci_server.public_ip
}

  description = "URL to access the LightCI Server web UI"
  value       = "http://${aws_instance.lightci_server.public_ip}:8080"

  description = "SSH command to connect to the EC2 instance"
  value       = "ssh -i <YOUR_KEY.pem> ec2-user@${aws_instance.lightci_server.public_ip}"
}

output "instance_id" {
  description = "EC2 instance ID"
  value       = aws_instance.lightci_server.id
}
```

## Error Scenarios and Handling

### Error: Default VPC does not exist
**Symptom**: `Error: no matching VPC found`.
**Generated fix**: Create a default VPC or replace the data source with an explicit VPC resource.

### Error: Key pair not found
**Symptom**: `InvalidKeyPair.NotFound`.
**Generated fix**: Create the key pair first:
```bash
aws ec2 create-key-pair --key-name lightci-key --query 'KeyMaterial' \
  --output text > ~/lightci-key.pem
chmod 400 ~/lightci-key.pem
```

### Error: Instance type not available in AZ
**Symptom**: `Your requested instance type is not supported in your requested Availability Zone`.
**Generated fix**: Change `availability_zone` in the subnet data source, or specify
`instance_type` as `t3.small` which has broader AZ availability.

### Error: Terraform state conflict
**Symptom**: Resources exist in AWS but not in state, or vice versa.
**Generated advisory**: AWSInfraAgent will describe how to use `terraform import` or
`terraform state rm` — but will not run these commands. Developer must execute.

## Important Notes

- AWSInfraAgent never sets `0.0.0.0/0` SSH access without a comment recommending
  restriction to the developer's IP in production.
- `terraform.tfstate` files are gitignored — never commit state to version control.
- For production, AWSInfraAgent recommends storing state in an S3 backend with
  DynamoDB locking.
