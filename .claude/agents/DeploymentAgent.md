# DeploymentAgent

## Purpose

DeploymentAgent generates complete SSH-based deployment scripts for the LightCI Server.
When invoked, it produces a `deploy.sh` script and a `docker-install.sh` script that
the developer runs manually against a live EC2 instance. DeploymentAgent never opens
SSH connections, never runs commands on remote servers, and never modifies infrastructure.

## Inputs

| Input | Type | Default | Description |
|---|---|---|---|
| `EC2_PUBLIC_IP` | env var | required | Public IP of the target EC2 instance |
| `SSH_KEY_PATH` | env var | required | Local path to the `.pem` SSH key file |
| `DOCKER_USERNAME` | env var | required | Docker Hub username |
| `DOCKER_IMAGE` | env var | `light-ci-server` | Image name |
| `DOCKER_TAG` | env var | `latest` | Image tag to deploy |
| `APP_PORT` | env var | `8080` | Container port to expose |
| `CONTAINER_NAME` | string | `light-ci-server` | Docker container name on EC2 |

---

## Outputs

| Output | Location | Description |
|---|---|---|
| Deployment script | `scripts/deploy.sh` | Full deployment script — run by developer |
| Docker install script | `scripts/docker-install.sh` | Standalone Docker setup for EC2 |
| Health check command | stdout | curl command to verify deployment |

---

## Step-by-Step Plan

### Step 1 — Validate inputs
DeploymentAgent verifies all required env vars are documented before generating scripts.
It flags missing variables and refuses to generate incomplete scripts.

### Step 2 — Generate docker-install.sh
A standalone script that installs Docker on Amazon Linux 2023 via SSH.
Developer runs this once when first setting up an EC2 instance (if Terraform
user_data hasn't already done it).

### Step 3 — Generate deploy.sh
The full deployment script:
1. Copy the script to the EC2 instance (optional, or run remotely via SSH)
2. Pull the Docker image
3. Stop and remove any existing container
4. Run the new container
5. Verify the container is healthy

### Step 4 — Output verification command
A curl command the developer can run locally after deploying to confirm
the app is responding.

## Generated Scripts

### scripts/docker-install.sh

```bash
#!/bin/bash
# Generated by DeploymentAgent — Docker Installation for EC2
# Run: bash scripts/docker-install.sh
# This script SSHes into EC2 and installs Docker (Amazon Linux 2023)
# DO NOT EXECUTE via Claude — review and run manually.

set -euo pipefail

EC2_PUBLIC_IP="${EC2_PUBLIC_IP:?EC2_PUBLIC_IP is required}"
SSH_KEY_PATH="${SSH_KEY_PATH:?SSH_KEY_PATH is required}"

echo "Installing Docker on EC2 at ${EC2_PUBLIC_IP}..."

ssh -i "${SSH_KEY_PATH}" \
    -o StrictHostKeyChecking=no \
    -o ConnectTimeout=30 \
    "ec2-user@${EC2_PUBLIC_IP}" << 'REMOTE_SCRIPT'

set -e
echo "[1/4] Updating packages..."
sudo dnf update -y

echo "[2/4] Installing Docker..."
sudo dnf install -y docker

echo "[3/4] Enabling and starting Docker service..."
sudo systemctl enable docker
sudo systemctl start docker

echo "[4/4] Adding ec2-user to docker group..."
sudo usermod -aG docker ec2-user

echo "Docker installation complete."
docker --version

REMOTE_SCRIPT

echo "Docker installed successfully on ${EC2_PUBLIC_IP}"
echo "Note: You may need to log out and back in for group changes to take effect."
```

### scripts/deploy.sh

```bash
#!/bin/bash
# Generated by DeploymentAgent — LightCI Server EC2 Deployment
# Run: bash scripts/deploy.sh
# Prerequisites: Docker installed on EC2, image pushed to Docker Hub
# DO NOT EXECUTE via Claude — review and run manually.

set -euo pipefail

# --- Configuration (set via environment or edit directly) ---
EC2_PUBLIC_IP="${EC2_PUBLIC_IP:?EC2_PUBLIC_IP is required}"
SSH_KEY_PATH="${SSH_KEY_PATH:?SSH_KEY_PATH is required}"
DOCKER_USERNAME="${DOCKER_USERNAME:?DOCKER_USERNAME is required}"
DOCKER_IMAGE="${DOCKER_IMAGE:-light-ci-server}"
DOCKER_TAG="${DOCKER_TAG:-latest}"
APP_PORT="${APP_PORT:-8080}"
CONTAINER_NAME="${CONTAINER_NAME:-light-ci-server}"
IMAGE_REF="${DOCKER_USERNAME}/${DOCKER_IMAGE}:${DOCKER_TAG}"

echo "============================================"
echo " LightCI Server — EC2 Deployment"
echo " Target:  ec2-user@${EC2_PUBLIC_IP}"
echo " Image:   ${IMAGE_REF}"
echo " Port:    ${APP_PORT}"
echo "============================================"

# --- Verify SSH key ---
if [ ! -f "${SSH_KEY_PATH}" ]; then
  echo "ERROR: SSH key not found: ${SSH_KEY_PATH}"
  exit 1
fi
chmod 400 "${SSH_KEY_PATH}"

# --- Deploy via SSH ---
echo "[1/4] Connecting to EC2 and deploying..."
ssh -i "${SSH_KEY_PATH}" \
    -o StrictHostKeyChecking=no \
    -o ConnectTimeout=30 \
    "ec2-user@${EC2_PUBLIC_IP}" \
    DOCKER_USERNAME="${DOCKER_USERNAME}" \
    DOCKER_IMAGE="${DOCKER_IMAGE}" \
    DOCKER_TAG="${DOCKER_TAG}" \
    APP_PORT="${APP_PORT}" \
    CONTAINER_NAME="${CONTAINER_NAME}" \
    IMAGE_REF="${IMAGE_REF}" \
    bash << 'REMOTE_SCRIPT'

set -e

echo "[2/4] Pulling Docker image: ${IMAGE_REF}"
docker pull "${IMAGE_REF}"

echo "[3/4] Stopping existing container (if any)..."
docker stop "${CONTAINER_NAME}" 2>/dev/null || true
docker rm "${CONTAINER_NAME}" 2>/dev/null || true

echo "[4/4] Starting new container..."
docker run -d \
  --name "${CONTAINER_NAME}" \
  --restart unless-stopped \
  -p "${APP_PORT}:${APP_PORT}" \
  -e SPRING_PROFILES_ACTIVE=prod \
  "${IMAGE_REF}"

echo "Container started. Waiting for health check..."
sleep 10

# Health check
HTTP_STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
  "http://localhost:${APP_PORT}/actuator/health" || echo "000")

if [ "${HTTP_STATUS}" = "200" ]; then
  echo "Health check PASSED (HTTP ${HTTP_STATUS})"
else
  echo "WARNING: Health check returned HTTP ${HTTP_STATUS}"
  echo "Container logs:"
  docker logs "${CONTAINER_NAME}" --tail 30
fi

docker ps --filter "name=${CONTAINER_NAME}"

REMOTE_SCRIPT

echo ""
echo "============================================"
echo " DEPLOYMENT COMPLETE"
echo " App URL: http://${EC2_PUBLIC_IP}:${APP_PORT}"
echo " Health:  http://${EC2_PUBLIC_IP}:${APP_PORT}/actuator/health"
echo "============================================"
echo ""
echo "Verify locally:"
echo "  curl http://${EC2_PUBLIC_IP}:${APP_PORT}/actuator/health"
```

## Error Scenarios and Handling

### Error: SSH connection refused
**Symptom**: `ssh: connect to host X.X.X.X port 22: Connection refused`.
**Generated fix**:
1. Verify the EC2 instance is running: check AWS Console or `aws ec2 describe-instances`.
2. Verify port 22 is open in the security group (check `terraform/main.tf`).
3. Verify the instance has finished booting (user_data takes 1-2 min).

### Error: Permission denied (publickey)
**Symptom**: `Permission denied (publickey)`.
**Generated fix**:
1. Verify `SSH_KEY_PATH` points to the correct `.pem` file.
2. Verify the key pair name in Terraform matches the `.pem` file you have locally.
3. Ensure the key file has permissions 400: `chmod 400 "${SSH_KEY_PATH}"`.

### Error: Docker not found on EC2
**Symptom**: `bash: docker: command not found` (remote).
**Generated fix**: Run `bash scripts/docker-install.sh` first, or re-provision
the EC2 instance if `user_data` failed.

### Error: Image pull failed
**Symptom**: `Error response from daemon: pull access denied`.
**Generated fix**:
1. The image must be pushed to Docker Hub before deploying. Run the docker-build step.
2. If the image is private, add Docker Hub login to the deploy script:
   ```bash
   docker login -u "${DOCKER_USERNAME}" -p "${DOCKER_PASSWORD}"
   ```

### Error: Port already in use
**Symptom**: `Bind for 0.0.0.0:8080 failed: port is already allocated`.
**Generated fix**: The old container stop/rm commands in deploy.sh handle this.
If a non-Docker process is using port 8080, identify it with `sudo lsof -i :8080`.

### Error: Health check fails after deploy
**Symptom**: HTTP 000 or non-200 from health endpoint.
**Generated fix**:
1. Check container logs: `docker logs light-ci-server --tail 50`
2. Verify `APP_PORT` matches the port in `application.properties`.
3. Spring Boot takes ~30s to start — increase the `sleep 10` in deploy.sh.

---

## Notes

- DeploymentAgent always generates scripts that stop the old container before
  starting the new one, ensuring zero port conflicts.
- The `--restart unless-stopped` Docker flag ensures the container restarts
  automatically after EC2 reboots.
- For production deployments, DeploymentAgent recommends using a health-check
  wait loop instead of a fixed `sleep`.
